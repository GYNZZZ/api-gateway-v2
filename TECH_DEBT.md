# API Gateway 技术债审计

## 总体判断

当前代码实现了有效的产品原型，但核心数据、鉴权、扣费和日志都集中在单个 `server.js` 与本地 JSON 文件中。继续叠加商用功能会快速放大一致性、安全和维护成本。建议先完成数据层与测试底座，再扩展注册、支付或多上游功能。

## JSON 文件存储的限制

### 当前实现

- `users.json` 保存用户、明文 API Key 和余额。
- `logs.json` 保存调用日志。
- 每次操作通过同步 `readFileSync` 读取完整文件。
- 写入时先写 `.tmp`，再使用 `renameSync` 覆盖。

### 限制与风险

- 同步文件 I/O 会阻塞 Node.js 事件循环。
- 每个请求读取或重写整个文件，数据量增长后延迟快速上升。
- 多个并发请求可能基于同一旧版本数据计算并互相覆盖。
- 扣余额与写日志不是同一事务，进程崩溃时可能只完成一半。
- 多实例部署会拥有各自文件副本，用户、余额和日志互不一致。
- Render/Railway 的临时文件系统可能在重启或重新部署后丢失数据。
- 无索引、分页、约束、备份、查询优化和迁移机制。
- `logs.json` 会无限增长，未授权攻击也能触发日志写入。

## 迁移 PostgreSQL 的必要性

PostgreSQL 不是性能优化项，而是内测前的数据正确性要求。

### 必须由数据库保证的能力

- 用户 API Key 唯一约束
- 用户、余额、充值流水和调用日志的关联关系
- 原子余额扣减，例如余额大于等于费用时才更新
- 扣费与调用记录的一致性事务
- 并发更新锁或条件更新
- 日志分页、时间范围和用户索引
- Schema migration、备份和恢复
- 多实例共享同一数据源

### 建议的数据模型

- `users`：用户状态、名称、创建时间
- `api_keys`：Key 哈希、前缀、状态、过期时间、最后使用时间
- `balances` 或账户余额字段
- `balance_transactions`：充值、扣费、退款、人工调整
- `request_logs`：请求 ID、用户、模型、状态、费用、耗时
- `admin_audit_logs`：操作者、目标、动作和变更内容

## 当前代码结构评估

### 当前状态

- 单个 `server.js` 约 300 行，包含配置、文件访问、鉴权、业务逻辑、上游调用、计费和路由。
- 静态页面使用独立 HTML/CSS/JS，适合当前无框架方案。
- 没有测试目录、服务层、数据访问层或统一配置校验。

### 是否现在拆分

不建议为了目录美观做大规模重构，但在 PostgreSQL 迁移时应按职责拆分，否则事务、测试和后续支付逻辑会难以控制。

### 建议边界

```text
src/
  app.js
  config.js
  routes/
    public.js
    user.js
    admin.js
  middleware/
    user-auth.js
    admin-auth.js
    error-handler.js
    rate-limit.js
  services/
    gateway-service.js
    billing-service.js
    user-service.js
  repositories/
    user-repository.js
    log-repository.js
  db/
    migrations/
```

保留 `server.js` 作为极薄的启动入口即可。

## 仍属于 MVP 的实现

- 固定每次调用扣费 `1`，没有按 Token 或模型计费。
- Mock 返回固定内容，usage 全部为 0。
- 上游仅支持一个 Base URL 和一个 API Key。
- 上游请求没有超时、取消、重试、熔断和流式代理。
- 管理员使用一个全局 Key，没有账户、角色和操作审计。
- 用户身份等同于长期 API Key，没有登录会话。
- 完整用户 Key 可被管理员接口读取。
- 日志无分页、筛选、归档和保留周期。
- 用户创建和充值无操作流水与幂等控制。
- 没有模型列表、配额、限流和风控。
- 错误处理只覆盖 JSON 解析错误和部分业务错误。
- 没有自动化测试和 CI 质量门槛。

## 具体技术债与建议

### 1. 余额并发一致性

**现状**：先读取余额，调用上游成功后重新读取并扣费。

**风险**：并发请求可能同时通过余额检查，产生超额消费；文件覆盖可能丢失更新。

**建议**：使用数据库条件更新和事务；引入不可变余额流水。

### 2. API Key 查询与存储

**现状**：遍历完整用户数组比较明文字符串。

**风险**：泄漏影响全部长期凭证，用户数量增长后查询变慢。

**建议**：Key 前缀定位记录，安全哈希校验，支持吊销与轮换。

### 3. 上游 HTTP 客户端

**现状**：直接调用原生 `fetch`，等待上游 JSON。

**风险**：无超时；上游返回非 JSON 时进入通用 502；不支持流式响应。

**建议**：封装 Gateway Service，设置 AbortSignal 超时，区分网络、协议和上游业务错误。

### 4. 日志写入

**现状**：每条日志重写完整 JSON 文件。

**风险**：阻塞、无限增长、并发覆盖和磁盘耗尽。

**建议**：数据库异步写入、索引、分页、保留策略和敏感字段白名单。

### 5. 输入校验

**现状**：使用手工条件判断，只校验少数字段。

**风险**：模型名、消息结构、余额精度和未知字段缺少明确约束。

**建议**：使用 Schema 校验并统一返回错误；金额使用整数最小单位或精确数值类型。

### 6. 配置管理

**现状**：环境变量在模块顶部读取，缺失时部分使用空字符串或默认值。

**风险**：生产环境可能带着 Mock 或弱管理员 Key 启动。

**建议**：启动时按环境校验所有必需配置，失败即退出；配置对象只读。

### 7. 测试与交付

**现状**：依赖人工启动和 HTTP 测试。

**风险**：鉴权隔离、扣费和错误处理容易在改动后回归。

**建议**：增加单元、集成、并发和安全测试；CI 执行 lint、test、依赖扫描和 secret scan。

## 推荐重构顺序

1. 先为现有行为建立集成测试，固定 API 合同。
2. 引入配置校验和统一错误处理中间件。
3. 设计 PostgreSQL Schema 与迁移脚本。
4. 抽出 Repository 和 Billing Service，以事务替代 JSON 更新。
5. 改造 API Key 哈希、吊销和轮换。
6. 增加限流、超时、安全头和监控。
7. 最后再开发注册、支付、多上游和复杂套餐。

## 暂时不建议做的重构

- 不为了技术栈升级而引入大型前端框架。
- 不在数据库迁移前继续扩展 JSON 文件字段和复杂账务逻辑。
- 不在计费事务稳定前接入真实支付。
- 不在监控与限流完成前开放公开注册。
