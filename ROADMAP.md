# API Gateway 产品路线图

## 当前定位

`api-gateway-v2` 目前是一个可演示、可本地运行、可部署到单实例 Node.js Web Service 的 AI API Gateway 原型。它已经打通了最小产品闭环，但数据安全、并发一致性、运维能力和商业合规尚未达到正式商用要求。

## 已完成功能

### Provider 与模型配置

- JSON 文件管理 Provider、模型和全局设置
- 管理员可新增、修改、启用或禁用 Provider 与模型
- 支持默认模型和维护模式
- 用户可通过 `GET /v1/models` 查询当前可用模型
- 当前仍为 Mock-only，不连接真实上游

### 产品与页面

- 官网首页：`GET /`
- API 文档页：`GET /docs`
- 用户中心：`GET /dashboard`
- 管理后台：`GET /admin`
- 统一的蓝紫色 SaaS 视觉风格及基础响应式适配
- Render Blueprint：`render.yaml`

### 用户与管理员能力

- 用户 API Key 鉴权，支持 Bearer Token 和 `x-api-key`
- 管理员 API Key 鉴权，支持 Bearer Token 和 `x-admin-api-key`
- 查询当前用户：`GET /v1/me`
- 查询本人日志：`GET /v1/logs`
- 管理员查看用户和日志
- 管理员创建用户、生成 API Key、手动充值
- 用户余额及成功调用固定扣费

### API Gateway 能力

- OpenAI-compatible `POST /v1/chat/completions`
- Mock 模式
- 上游 API 转发模式
- 上游失败不扣费
- 调用状态、模型、扣费、余额和耗时日志
- API Key 脱敏日志
- 健康检查：`GET /health`

## 距离 MVP 内测版还缺什么

MVP 内测版应限定为少量受信任用户、单实例运行、人工开户和人工充值。进入内测前至少需要完成：

1. 将用户、余额和日志迁移到 PostgreSQL。
2. API Key 不再明文存储，改为哈希校验，并仅在创建时展示一次。
3. 建立原子扣费事务，避免并发请求导致余额重复消费或覆盖。
4. 增加全局、用户级和 IP 级限流。
5. 增加基础安全中间件、请求超时、统一错误处理和输入校验。
6. 增加自动化测试，覆盖鉴权、数据隔离、扣费和上游失败。
7. 移除仓库中真实可用的测试用户 Key，并轮换已经提交过的 Key。
8. 建立备份、恢复、日志保留和服务监控方案。
9. 明确内测规则、服务支持渠道和故障处理方式。

## 距离正式商用版还缺什么

- 正式账户体系：注册、登录、邮箱验证、密码重置、会话管理。
- 支付与账务：订单、支付回调、充值流水、退款、对账和发票。
- 精确计费：按模型、输入输出 Token、上游成本和套餐计费。
- PostgreSQL 高可用、Redis 限流与缓存、任务队列。
- 多上游供应商、故障转移、模型路由和熔断降级。
- API Key 生命周期：创建、轮换、吊销、过期、权限范围和最后使用时间。
- 多管理员角色与 RBAC，不再依赖单个全局管理员 Key。
- 完整可观测性：指标、结构化日志、Tracing、告警和状态页。
- 隐私政策、用户协议、退款规则、数据保留政策和合规评估。
- 安全测试、依赖漏洞扫描、密钥扫描和事件响应流程。
- 容量规划、SLA、灰度发布、回滚、灾难恢复和客户支持。

## 分阶段开发计划

### P0：安全与数据底座

**目标**

让系统达到小规模受控内测的最低安全和一致性标准。

**功能**

- PostgreSQL 用户、余额、日志和充值流水表
- 数据库事务扣费及幂等请求设计
- API Key 哈希存储、前缀索引、吊销和轮换
- 环境变量校验，生产环境禁止弱管理员 Key
- 限流、请求超时、请求体大小限制和安全响应头
- 自动化集成测试和 CI

**主要风险**

- 数据迁移可能造成余额或用户 ID 不一致
- Key 哈希改造会影响现有测试 Key
- 事务设计错误会直接造成账务损失

### P1：内测运营能力

**目标**

支持少量真实用户稳定使用，并能定位和处理问题。

**功能**

- 管理员操作审计日志
- 用户禁用、Key 吊销、余额流水
- 日志分页、筛选和保留周期
- 服务指标、错误告警、上游延迟监控
- 数据备份和恢复演练
- 内测说明、联系方式和状态通知

**主要风险**

- 审计数据增长导致查询变慢
- 告警阈值不合理产生噪音或漏报
- 人工充值仍可能出现操作错误

### P2：计费与商业闭环

**目标**

建立可核对、可追踪的真实计费和支付流程。

**功能**

- 模型价格表和 Token 用量计费
- 账单、充值订单、支付回调和对账
- 套餐、余额提醒、消费明细
- 用户协议、隐私政策、退款和发票规则
- 风控规则和异常消费冻结

**主要风险**

- 上游用量与本地计费不一致
- 支付回调重放或伪造
- 价格变更、退款和税务处理复杂

### P3：正式商用与规模化

**目标**

支持多实例、高可用和可承诺的服务质量。

**功能**

- Redis 分布式限流与缓存
- 多上游路由、熔断、重试和故障转移
- RBAC、团队账户和细粒度权限
- 灰度发布、自动回滚和灾难恢复
- SLA、公开状态页和客户支持流程
- 安全审计与渗透测试

**主要风险**

- 分布式系统增加一致性和运维复杂度
- 重试策略可能造成重复扣费或上游成本放大
- SLA 和合规承诺带来法律与运营责任

## MVP 内测准入标准

- PostgreSQL 迁移完成，余额扣费有事务保障
- 不存在仓库内明文可用密钥
- API Key 可吊销，管理员凭证可轮换
- 鉴权、数据隔离、扣费和上游失败测试全部通过
- 限流、超时、监控、备份和告警可用
- 明确内测人数、服务范围和数据保留规则
